# パッチノート作成ルール（内部用 / Codex向け）

このドキュメントは、`docs/patch_notes_public.md` を編集するときのルールです。  
公開画面（ホーム）には表示しません。

## 1. バージョンと並び順
- 見出しは `## vMAJOR.MINOR.PATCH（YYYY-MM-DD）` の形式にする（例: `## v0.3.0（2026-01-26）`）。
- 並び順は **上が最新（新しい日付/新しいバージョン）**。
- **日付が新しいほどバージョンも新しい**（逆転させない）。

## 2. カテゴリ構成（基本形）
各バージョンは、まずカテゴリ（箇条書き）を書き、次に詳細をネストする。

- 新カード追加
- カード調整
- ロール調整
- UI調整
- 機能追加（必要なときだけ）

カテゴリが不要なら省略してOKだが、**同種の内容が複数ある場合はカテゴリを作る**。

## 3. ロール追加の書き方
ロールを追加した場合は、以下の形式で書く（例）。

- 新ロールを追加
  - ［効率］HP.１２　Atk.２　Def.１　Spe.３　Bra.２
    - アイテムの効果が２倍になる

ポイント:
- ロール名は `［ ］` で囲む。
- ステータスは `HP.１２　Atk.２　Def.１　Spe.３　Bra.２` のように **全角スペース区切り**で統一。
- 効果説明は `-` で1行（長い場合のみ複数行）。

## 4. ロール調整の書き方
ロール能力を変更した場合は、以下の形式で書く（例）。

- ロール調整
  - ［延期］：ラウンド中に受けたダメージは追加ターンにまとめて受けるように

ポイント:
- **変更が入ったロールは全部**、このブロックに列挙する。
- 文章は「何がどう変わったか」が1行で分かるようにする。

## 5. 新カード追加の書き方
カード追加は、カード名（種別）と効果を1〜2行で書く。

- 新カード追加
  - 「再構築」（呪文）: 手札を2枚捨て、その枚数分山札から引く。

ポイント:
- カード名は `「 」`、装備カード名は `【 】` で統一。
- 種別は `（攻撃/呪文/装備/設置...）` のように括弧で付ける。

## 6. 公開用として避けること
- 実装の内部事情（ファイル名、関数名、APIパスなど）は原則書かない（必要なら `docs/progress.md` へ）。
