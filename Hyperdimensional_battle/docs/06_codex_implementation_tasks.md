# 超次元バトル - Codex向け実装タスク分解

## 基本方針
この文書の目的は、Codex に段階的に実装させるための作業単位を明確化すること。

## Phase 1: リポジトリ初期構成
### タスク
- monorepo 構成を作る
- `apps/web`, `packages/engine`, `packages/shared`, `workers/room-worker` を用意する
- TypeScript / ESLint / Prettier / Vitest を導入する
- Vite + React のWebアプリを作る
- Cloudflare Workers 開発設定を追加する

### 完了条件
- 開発サーバが起動する
- engine パッケージを web から import できる
- worker がローカル起動できる

## Phase 2: 共通型とデータ定義
### タスク
- 共有DTO型を作る
- CardDefinition / RoleDefinition / ReplayEvent / RoomState を定義する
- zod 等でバリデーションを追加する（任意だが推奨）
- サンプル役職とサンプルカードをJSONで置く

### 完了条件
- サンプルデータを読み込める
- 型エラーなくビルドできる

## Phase 3: ゲームエンジン基盤
### タスク
- GameState / PlayerState を実装する
- ラウンド進行の state machine を作る
- ドロー、マリガン、配置、解決、最終攻撃、ラウンド終了を実装する
- ログ生成機構を作る

### 完了条件
- ヘッドレスで5ラウンド完走できる
- ログが出る

## Phase 4: 効果解決エンジン
### タスク
- EffectDefinition / OperationDefinition を解釈する仕組みを作る
- 基礎 / 一時ステータス更新
- 破壊 / 無効 / 作成 / エンチャント付与
- 連続 / 連結判定
- 自己再発動ルールと他カード再発動ルール
- ループガード

### 完了条件
- サンプルカード（パンチ、はどうだん、筋トレ、ハイブリッド、焼却処分など）が正しく動作する
- リプレイイベント列を出力できる

## Phase 5: ローカルUI
### タスク
- 一画面ゲームUIを作る
- 手札表示
- 場表示
- ステータス表示
- ログ表示
- ラウンド進行UI
- カード詳細ポップアップ

### 完了条件
- ローカルで1人分の操作と再生ができる

## Phase 6: 演出システム
### タスク
- PixiJS を導入する
- CARD_ACTIVATED / DAMAGE_DEALT / FINAL_ATTACK などに対応する演出レイヤーを作る
- Howler.js を導入する
- 再生テンポ加速ロジックを実装する

### 完了条件
- 発動順に応じてエフェクトと音が鳴る
- 後半ほどテンポが短くなる
- UI視認性を損なわない

## Phase 7: ルーム同期
### タスク
- Worker ルーティングを実装する
- Durable Object に RoomState を持たせる
- JOIN / READY / START / SUBMIT / REPLAY 制御を実装する
- WebSocket 接続管理を実装する
- 再接続時の再同期を実装する

### 完了条件
- 複数クライアントが同じルームに参加できる
- 同一の再生状態を共有できる

## Phase 8: 共有観戦
### タスク
- 各プレイヤーの結果イベント列を順番に再生する
- 現在再生中プレイヤーを UI で強調する
- 全員の画面が同じ位置で進むことを保証する
- ラウンド結果 / 最終結果を表示する

### 完了条件
- ルーム内全員で同一の観戦ができる

## Phase 9: テスト
### タスク
- engine のユニットテスト
- サンプルカード回帰テスト
- 共有観戦同期テスト
- 再接続復元テスト

### 完了条件
- 主要ルールと同期ロジックが自動テストで確認できる
